# Habit Tracker PWA

## Overview

Single-page Progressive Web App for tracking daily habits and habit groups. Fully offline-capable, installable on mobile, all data in `localStorage`. Deployed to GitHub Pages.

## Project Structure

```
habit_tracking/
├── index.html       # Full app (HTML + CSS + JS inline)
├── sw.js            # Service Worker (cache-first, version: habit-tracker-v1.13)
├── manifest.json    # PWA manifest (standalone, portrait, purple theme)
├── icon-192.png     # App icon 192×192
├── icon-512.png     # App icon 512×512
└── README.md
```

## Architecture

- **Single file app**: All HTML, CSS, JS in `index.html` — no build tools, no frameworks
- **localStorage**: Key `habit_tracker_data`, stores `{ settings, items[] }` as JSON
- **Service Worker** (`sw.js`): Cache-first. Cache name: `habit-tracker-v1.13` — bump to force refresh
- **PWA manifest**: Standalone display, portrait, theme `#6C3CE1`, bg `#0f0f1a`

## Data Model

```json
{
  "settings": { "resetHour": 2 },
  "items": [
    { "id": "...", "type": "habit", "name": "...", "completed": ["YYYY-MM-DD"], "activeDays": [0,1,2,3,4,5,6] },
    {
      "id": "...", "type": "group", "name": "...", "completed": ["YYYY-MM-DD"], "activeDays": [0,1,2,3,4,5,6],
      "children": [{ "id": "...", "name": "...", "completed": ["YYYY-MM-DD"] }]
    }
  ]
}
```

- **`activeDays`**: Array of weekday numbers (0=Sun…6=Sat). Default `[0,1,2,3,4,5,6]`. Items only appear on active days. Missing/empty = all days (backward compat)
- **Group `completed`**: dates where master checkbox was tapped
- **Child `completed`**: individual completion dates
- Old format (`{ habits: [...] }`) auto-migrates on first load

## ⏰ Reset Hour — Full Semantics

`settings.resetHour` (0–23) defines when the logical "day" starts. Default is `0` (midnight).

### How it works

The "logical day" is the window from `resetHour` on one calendar day to `resetHour` on the next. The app stores all completions as the **calendar date at the start of that window** (i.e., the calendar date returned by `getEffectiveDate()` at the moment of completion).

**`getEffectiveDate(settings)`** — returns the calendar date considered "today":
- If `now.getHours() < resetHour` → returns **yesterday's** calendar date string
- Otherwise → returns **today's** calendar date string

### Example: resetHour = 18 (6 PM)

The logical "Saturday" runs from **Friday 6:00 PM → Saturday 5:59 PM**.

| Wall clock (real time) | `getEffectiveDate()` returns | Logical day | `activeDays` weekday |
|---|---|---|---|
| Friday 5 PM (< 6 PM) | Thursday's date | Friday's logical day | Friday (stored Thu+1) |
| **Friday 7 PM (≥ 6 PM)** | **Friday's date** | **Saturday's logical day** | **Saturday (stored Fri+1)** |
| Saturday 11 AM (< 6 PM) | Friday's date | **Saturday's logical day** | **Saturday (stored Fri+1)** |
| Saturday 7 PM (≥ 6 PM) | Saturday's date | Sunday's logical day | Sunday (stored Sat+1) |

> **Key insight**: With `resetHour = 18`, a "Saturday habit" (`activeDays = [6]`) is **stored under the Friday calendar date**. The Sunday-activeDays habit is stored as Saturday's date, etc. Each habit's stored effective date is always 1 calendar day *behind* its logical weekday when `resetHour > 13`.

### The +1 offset rule (`resetHour > 13`)

`isItemActiveOnDate(item, dateStr, settings)` checks which logical weekday a stored date belongs to:

```js
const d = new Date(dateStr + 'T12:00:00');
if (resetHour > 13) d.setDate(d.getDate() + 1);  // shift +1 day to get logical weekday
return days.includes(d.getDay());
```

This means stored date `"2026-03-06"` (Friday) with `resetHour=18` → checks `getDay()` of Saturday March 7 → returns `6` → matches `activeDays=[6]` (Saturday habit) ✓

### ⚠️ CRITICAL: Never use raw `getDay()` against `activeDays`

**Always** use `effectiveDayOfWeek(dateStr, settings)` or `isItemActiveOnDate()` instead of `date.getDay()` whenever comparing stored dates against `activeDays`. Raw `getDay()` will be wrong for `resetHour > 13`.

| Situation | Use |
|---|---|
| Check if a stored date is an active day | `effectiveDayOfWeek(dateStr, settings)` |
| Check if an item is active on a stored date | `isItemActiveOnDate(item, dateStr, settings)` |
| Convert physical calendar cell → stored effective date | `calendarDateToEffectiveDate(calDateStr, settings)` |
| Toggle/read from heatmap calendar cell | Convert with `calendarDateToEffectiveDate()` first |

### Calendar heatmap display

The heatmap iterates physical calendar dates (Sun–Sat columns). For each cell:
- Physical cell date → `calendarDateToEffectiveDate()` → effective date key
- Check `completedDates.has(effectiveDate)` → filled/green
- Check `effectiveDate === todayStr` → today highlight
- Check `effectiveDate > todayStr` → future (blank)

This means Saturday's cell shows green when the stored Friday effective date is in `completedDates`.

### `calendarDateToEffectiveDate(calDateStr, settings)`

Converts a physical calendar cell date to the effective date that would be stored:
- `resetHour <= 13`: returns `calDateStr` unchanged
- `resetHour > 13`: returns `calDateStr - 1 day`

Example: clicking Saturday `"2026-03-07"` with `resetHour=18` → stores `"2026-03-06"` (Friday)

## Key Features

| Feature | Implementation |
|---|---|
| Habit Groups | `type: "group"` items with `children[]` array |
| Master checkbox | Toggles all children + group `completed` |
| Auto-complete group | When all children checked, group auto-completes |
| Auto-hide completed | Completed items get `hiding` CSS class; toggle via "Show completed" button |
| All-done celebration | Shown when all items done and `showCompleted` is false |
| Detail view | Fullscreen overlay with stats cards, 3-month heatmap (most recent first), per-child breakdown. Tapping group children opens their own detail view. |
| Day-of-week scheduling | `activeDays` array on items; day picker in add/edit modal; items hidden on non-active days; streaks skip inactive days |
| Custom reset hour | `getEffectiveDate()` checks `settings.resetHour` vs current hour; full offset semantics for weekly habits — see Reset Hour section above |
| Reorder mode | ↕️ button toggles `reorderMode`; shows ▲/▼ arrows on items; `moveItem()` swaps in data array |
| Edit mode | ✏️ button opens add modal pre-filled with item data; `editingId` tracks which item is being edited |
| Streak (habit) | `calcStreak()` — consecutive active-day completions, reset-hour aware |
| Streak (group) | `calcGroupStreak()` — consecutive days all children done, reset-hour aware |
| Perfect day streak | `calcPerfectDayStreak()` — consecutive days every item done |
| Settings modal | Gear icon in header, saves `resetHour` |
